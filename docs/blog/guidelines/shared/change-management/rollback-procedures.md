# ロールバック手順

> **目的**: コンテンツ変更の問題発生時に迅速かつ安全的に元に戻すための手順
> **更新日**: 2025-04-11
> **バージョン**: 1.0

---

## 概要

本ドキュメントは、ブログコンテンツの変更によって問題が発生した場合に、コンテンツを以前の状態に戻す（ロールバック）ための手順を定義します。

---

## ロールバックの種類

### 種類1: 記事の非公開

**目的**: 記事を一時的に非公開にする

**手順**:
```sql
-- 1. 記事のステータスを draft に変更
UPDATE blog_posts 
SET status = 'draft',
    updated_at = NOW()
WHERE slug = '該当スラッグ';

-- 2. 確認
SELECT title, slug, status, updated_at
FROM blog_posts
WHERE slug = '該当スラッグ';
```

**確認事項**:
- [ ] 記事が非公開になったことを確認
- [ ] 404エラーが返ってくることを確認
- [ ] サイトマップから削除されたことを確認

### 種類2: コンテンツの復元

**目的**: 以前のバージョンに戻す

**手順**:
```bash
# 1. Gitで履歴を確認
git log --oneline --follow docs/blog/articles/[ファイル名]

# 2. 以前のバージョンを確認
git show [コミットハッシュ]:docs/blog/articles/[ファイル名]

# 3. 以前のバージョンに復元
git checkout [コミットハッシュ] -- docs/blog/articles/[ファイル名]

# 4. データベースを更新
-- Markdownの内容を更新
UPDATE blog_posts 
SET content_markdown = '[復元したMarkdown]',
    content_html = '[HTMLに変換した内容]',
    updated_at = NOW()
WHERE slug = '該当スラッグ';
```

**確認事項**:
- [ ] コンテンツが正しく復元されたことを確認
- [ ] 表示が正しいことを確認
- [ ] リンクが正常に機能することを確認

### 種類3: 複数記事の一括非公開

**目的**: 複数の記事を一時的に非公開にする

**手順**:
```sql
-- 1. 複数の記事を非公開に変更
UPDATE blog_posts 
SET status = 'draft',
    updated_at = NOW()
WHERE slug IN ('article-1', 'article-2', 'article-3');

-- 2. 確認
SELECT title, slug, status, updated_at
FROM blog_posts
WHERE slug IN ('article-1', 'article-2', 'article-3');
```

## 緊急ロールバック

### トリガー

緊急ロールバックが必要な状況：
- 重大な誤情報が発見された
- SEOに悪影響が出ている
- 技術的な問題が発生した
- 法的な問題が発生した

### 手順

#### ステップ1: 即時対応

```sql
-- 1. 該当記事を即座に非公開にする
UPDATE blog_posts 
SET status = 'draft',
    updated_at = NOW()
WHERE slug = '問題のある記事のスラッグ';

-- 2. 複数の記事がある場合は一括で対応
UPDATE blog_posts 
SET status = 'draft',
    updated_at = NOW()
WHERE slug IN ('article-1', 'article-2');
```

#### ステップ2: 影響評価

```sql
-- 1. 過去24時間のページビューを確認
SELECT slug, title, view_count, updated_at
FROM blog_posts
WHERE updated_at >= NOW() - INTERVAL '24 hours'
ORDER BY view_count DESC;

-- 2. 検索トラフィックを確認
-- （Google Analytics、Naver Analyticsで確認）
```

#### ステップ3: 通知

- ステークホルダーに通知
- 原因と対策を報告
- 再公開の目処を共有

## 計画的ロールバック

### トリガー

計画的ロールバックが必要な状況：
- A/Bテスト結果に基づく判断
- 戦略的な方向転換
- 定期的なコンテンツ更新

### 手順

#### ステップ1: 事前準備

1. 新しいコンテンツを準備
2. ステークホルダーの承認を取得
3. 公開スケジュールを設定

#### ステップ2: 実行

```sql
-- 1. 旧コンテンツをアーカイブ
UPDATE blog_posts 
SET status = 'archived',
    updated_at = NOW()
WHERE slug = '旧記事のスラッグ';

-- 2. 新しいコンテンツを公開
UPDATE blog_posts 
SET status = 'published',
    published_at = NOW(),
    updated_at = NOW()
WHERE slug = '新記事のスラッグ';
```

#### ステップ3: 事後処理

- リダイレクト設定（必要な場合）
- サイトマップの更新
- 検索エンジンに再インデックスをリクエスト

## ロールバック後の確認事項

### 技術的確認

- [ ] 記事が正しく非公開/復元された
- [ ] 表示が正しい
- [ ] リンクが正常に機能する
- [ ] モバイル表示が正しい

### SEO確認

- [ ] 検索エンジンで非公開/復元を確認
- [ ] サイトマップが更新された
- [ ] インデックス状況を確認

### ユーザー確認

- [ ] ユーザーからの問い合わせがないか
- [ ] アクセスログに異常がないか
- [ ] エラーログに異常がないか

## ロールバック防止策

### 事前の対策

1. **テスト環境での確認**
   - 本番環境に適用前にテスト環境で確認
   - ステージング環境で動作確認

2. **バックアップの作成**
   - 変更前にバックアップを作成
   - バージョン管理システムで追跡

3. **段階的な公開**
   - 一部の記事のみ先行公開
   - 問題がないことを確認してから全記事公開

### 事中の対策

1. **監視**
   - 公開直後のアクセスを監視
   - エラーログを監視

2. **即時対応体制**
   - 問題発生時に即時対応できる体制
   - 連絡チャンネルの明確化

## ロールバック後の改善

### 原因究明

1. 問題の根本原因を特定
2. 再発防止策を検討
3. プロセス改善を実施

### ドキュメント更新

1. 問題の内容を記録
2. 対応内容を記録
3. 教訓を共有
