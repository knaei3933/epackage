# 変更伝播プロトコル

> **目的**: コンテンツ変更を日本語・韓国語両方に伝播し、一貫性を維持するためのプロトコル
> **更新日**: 2025-04-11
> **バージョン**: 1.0

---

## 概要

本プロトコルは、Epackage Labブログコンテンツの変更を日本語と韓国語の両方に伝播し、ブランドの一貫性とSEO効果を最大化するための手順を定義します。

## 変更タイプ分類

### タイプ1: Core変更（共通コンテンツ）

**定義**: ブランド、製品情報、会社情報など、すべての言語で一貫性が必要な変更

**例**:
- 会社情報の更新（所在地、連絡先）
- 製品仕様の変更（最小ロット、納期、価格）
- ブランドガイドラインの変更
- 法規制対応（食品表示法など）

**伝播プロセス**:
1. Coreコンテンツを更新（日本語）
2. 韓国語翻訳を依頼
3. 両言語で同時公開

### タイプ2: Language-specific変更（言語固有）

**定義**: 特定の言語市場に固有な変更

**例**:
- 日本市場のSEO対策
- 韓国市場のトレンド対応
- 現地の法規制対応
- 現地のキャンペーン連携

**伝播プロセス**:
1. 該当言語でのみ変更を実施
2. 他言語には影響なし

### タイプ3: 分離変更（Independent）

**定義**: 各言語で独立したコンテンツ戦略

**例**:
- 現地固有の事例紹介
- 現地のパートナーシップ記事
- 現地のイベントレポート

**伝播プロセス**:
1. 各言語で独立して作成
2. 共通項目のみ同期

## 変更ワークフロー

### フェーズ1: 変更リクエスト

1. **変更提案**
   - 変更内容を文書化
   - 影響範囲を特定（Core/Language-specific/分離）
   - 優先度を設定（高/中/低）

2. **レビュー依頼**
   - ステークホルダーにレビューを依頼
   - SEO影響を評価
   - 翻訳コストを見積もり

### フェーズ2: 実装

1. **Core変更の場合**
   ```
   [日本語] コンテンツ更新 → レビュー → 承認
        │
        ▼
   [韓国語] 翻訳依頼 → 翻訳レビュー → 承認
        │
        ▼
   [両言語] 同時公開
   ```

2. **Language-specific変更の場合**
   ```
   [該当言語] コンテンツ更新 → レビュー → 承認 → 公開
   ```

3. **分離変更の場合**
   ```
   [日本語] 独自コンテンツ作成 → レビュー → 承認 → 公開
   [韓国語] 独自コンテンツ作成 → レビュー → 承認 → 公開
   ```

### フェーズ3: 品質確認

1. **公開前チェック**
   - [ ] 両言語のリンク切れ確認
   - [ ] 画像の表示確認
   - [ ] 構造化データのテスト
   - [ ] モバイル表示の確認

2. **公開後モニタリング**
   - [ ] インデックス状況の確認
   - [ ] 検索順位のモニタリング
   - [ ] ユーザーエンゲージメントの追跡

## ロールバック手順

### 緊急ロールバック

**トリガー**:
- 重大な誤情報が発見された
- SEOに悪影響が出ている
- 技術的な問題が発生した

**手順**:
1. **即時対応**
   ```bash
   # 1. 該当記事のステータスを draft に変更
   UPDATE blog_posts 
   SET status = 'draft' 
   WHERE slug = '該当スラッグ';
   
   # 2. 過去のバージョンから復元
   # （バージョン管理システムから復元）
   ```

2. **影響評価**
   - 影響を受けた記事数を確認
   - 検索順位への影響を評価
   - ユーザーへの影響を確認

3. **通知**
   - ステークホルダーに通知
   - 原因と対策を報告

### 計画的ロールバック

**トリガー**:
- A/Bテスト結果に基づく判断
- 戦略的な方向転換

**手順**:
1. **事前準備**
   - ロールバック計画を策定
   - ステークホルダーの承認を取得

2. **実行**
   - 新しいコンテンツを準備
   - 両言語で同時に公開

3. **事後処理**
   - 旧コンテンツのアーカイブ
   - リダイレクト設定（必要な場合）

## 変更管理のベストプラクティス

### 変更の batching
- 小さな変更をまとめて実施（頻度を減らす）
- 四半期ごとの大規模更新を検討

### バージョン管理
- すべての変更をバージョン管理システムで追跡
- 変更理由をコミットメッセージに明記

### コミュニケーション
- 変更予定をステークホルダーに共有
- 変更履歴を記録

## 緊急時の連絡チャネル

| 緊急度 | 連絡チャネル | 応答時間 |
|-------|-------------|----------|
| **緊急** | 即時メッセージ（Slack/Teams） | 30分以内 |
| **高** | メール + 即時メッセージ | 2時間以内 |
| **中** | メール | 1営業日以内 |
| **低** | プロジェクト管理ツール | 次回更新時 |
