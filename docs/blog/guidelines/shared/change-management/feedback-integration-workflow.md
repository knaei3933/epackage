# フィードバック統合ワークフロー

> **目的**: 専門家レビュアーからのフィードバックを効率的に統合し、コンテンツ品質を向上させるためのプロセス
> **更新日**: 2025-04-11
> **バージョン**: 1.0

---

## 概要

本ワークフローは、専門家レビュアー（技術、法務、SEO）からのフィードバックを収集、評価、統合し、最終的なコンテンツ品質を保証するためのプロセスを定義します。

---

## フィードバック収集

### 収集方法

#### 1. レビューフォーム

**使用ツール**:
- Google Forms
- Typeform
- 社内プロジェクト管理ツール

**フォーム項目**:
```markdown
# コンテンツレビューフォーム

## 記事情報
- タイトル: [記事タイトル]
- URL: [記事URL]
- カテゴリー: [カテゴリー]
- 作成者: [作成者名]

## レビュー項目

### 技術的な正確性
- 評価: [問題なし / 修正必要 / 大きな問題あり]
- コメント: [具体的な指摘事項]

### 法的リスク
- 評価: [問題なし / 修正必要 / 大きな問題あり]
- コメント: [具体的な指摘事項]

### SEO最適化
- 評価: [問題なし / 改善余地あり / 大きな問題あり]
- コメント: [具体的な改善提案]
```

#### 2. 直接コメント

**使用ツール**:
- Google Docsのコメント機能
- GitHubのPull Requestレビュー
- Figmaのコメント機能

#### 3. 会議でのフィードバック

**形式**:
- 定例ミーティングでの議論
- アドホックな会議での確認
- 電話会でのレビュー

## フィードバック分類

### 優先度分類

| 優先度 | 定義 | 対応期限 |
|-------|------|----------|
| **高** | 重大な誤情報、法的リスク、SEO致命的な問題 | 24時間以内 |
| **中** | 技術的な不正確、SEO改善余地、表現の調整 | 1週間以内 |
| **低** | 文体の微調整、フォーマットの統一 | 次回更新時 |

### カテゴリー分類

| カテゴリー | 具体的な内容 |
|---------|----------------|
| **技術的正確性** | 製品仕様、技術用語、製造プロセス |
| **法的コンプライアンス** | 商標、著作権、法規制 |
| **SEO** | キーワード、メタデータ、構造化データ |
| **品質** | 文体、フォーマット、整合性 |

## 統合プロセス

### フェーズ1: フィードバック収集

1. **フィードバックの収集**
   - レビューフォームからの回答を収集
   - 直接コメントを集約
   - 会議での議論を記録

2. **フィードバックの整理**
   ```markdown
   # フィードバック整理シート
   
   ## 記事: [タイトル]
   
   | カテゴリー | 優先度 | 内容 | 担当者 |
   |-----------|--------|------|--------|
   | 技術 | 高 | 最小ロットが「1個から」となっているが、正しくは「500枚から」 | [担当者] |
   | 法務 | 中 | 競合他社との比較表現を控えめにする必要がある | [担当者] |
   | SEO | 低 | キーワード密度を2%に調整すると効果が上がる可能性がある | [担当者] |
   ```

### フェーズ2: 優先度評価

1. **影響度評価**
   - ビジネスへの影響
   - SEOへの影響
   - ブランドへの影響

2. **対応順序の決定**
   - 高優先度の項目から対応
   - 依存関係がある場合は並行して対応

### フェーズ3: 修正実施

1. **修正の実施**
   - フィードバックに基づいて修正を実施
   - 修正内容を記録

2. **修正内容の確認**
   ```sql
   -- 修正履歴の記録
   INSERT INTO content_revisions (
     post_id,
     revision_number,
     revision_comment,
     revised_by,
     revised_at
   ) VALUES (
     [post_id],
     [revision_number],
     '[修正内容の要約]',
     '[修正者]',
     NOW()
   );
   ```

### フェーズ4: 再レビュー

1. **修正内容の確認**
   - レビュアーが修正内容を確認
   - 承認または再修正の指示

2. **最終承認**
   - すべてのレビュアーが承認
   - 公開の承認を取得

## 品質指標

### フィードバック対応時間

| 指標 | 目標値 | 測定方法 |
|------|--------|----------|
| **高優先度対応時間** | 24時間以内 | フィードバック収集から修正完了まで |
| **中優先度対応時間** | 1週間以内 | フィードバック収集から修正完了まで |
| **低優先度対応時間** | 次回更新時 | 次回の更新に含めて対応 |

### フィードバック品質

| 指標 | 目標値 | 測定方法 |
|------|--------|----------|
| **フィードバック率** | 100% | すべてのコンテンツがレビューされている |
| **修正承認率** | 95%以上 | 最初の修正で承認される割合 |
| **再修正率** | 10%以下 | 再修正が必要な割合 |

## フィードバックの記録と活用

### 記録

```sql
-- フィードバック履歴の記録
CREATE TABLE feedback_history (
    id SERIAL PRIMARY KEY,
    post_id UUID REFERENCES blog_posts(id),
    reviewer_id UUID REFERENCES users(id),
    category TEXT NOT NULL, -- 'technical', 'legal', 'seo', 'quality'
    priority TEXT NOT NULL, -- 'high', 'medium', 'low'
    feedback TEXT NOT NULL,
    status TEXT NOT NULL, -- 'pending', 'in_progress', 'resolved', 'deferred'
    created_at TIMESTAMP DEFAULT NOW(),
    resolved_at TIMESTAMP
);
```

### 活用

1. **パターンの分析**
   - よくある問題の特定
   - プロセス改善のためのインプット

2. **トレーニング資料**
   - よくある問題の回避方法
   - ベストプラクティスの共有

3. **ガイドラインの更新**
   - 新しい問題への対応
   - 既存ガイドラインの改善
